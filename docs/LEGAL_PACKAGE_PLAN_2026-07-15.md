## 3. Предварительные решения 0.1–0.6 — статус по проекту

Без закрытия этих пунктов тексты оферты/Политики писать нельзя (переделка будет тотальной). Ниже — **рекомендация по факту продукта** + статус `OPEN` до вашей финальной версии.

### 0.1. Круг лиц: B2B или B2C — **OPEN**

| Факт в продукте | Риск |
|-----------------|------|
| Self-serve регистрация без проверки «юрлицо/ИП» | Формально открыт доступ физлицам → ЗоЗПП |
| Гостевые демо Talk/Voice на лендинге (60 с, без аккаунта) | Маркетинговая обработка + B2C-adjacent |
| Позиционирование — Digital Workforce OS / Enterprise | Согласуется с B2B-only офертой |

**Рекомендация (из структуры):** оферта **только юрлицам и ИП**; при регистрации — подтверждение статуса; физлицам доступ к платной платформе не предоставляется.  
**Отдельно решить:** гостевое демо = не акцепт оферты, а публичная пробная демонстрация с урезанной политикой / согласием cookie + кратким дисклеймером ИИ.

→ В финальной версии нужно: `B2B_ONLY = yes/no` + правила для landing demo.

### 0.2. Оператор vs обработчик — **OPEN (модель ясна, тексты нет)**

| Поток | Роль NULLXES | Где описать |
|-------|-------------|-------------|
| Лиды, аккаунты админов, посетители сайта, кандидаты | **Оператор** | Политика ПДн (док. 3) |
| ПД сотрудников клиента, БЗ, диалоги Talk, миссии | **Обработчик по поручению** | DPA (док. 9) + отсылка в оферте §14 |

Сейчас `/docs/personal-data` называет NULLXES **только оператором** — это **неверно** для B2B SaaS с контентом клиента. Исправить при написании.

В коде уже есть: журнал согласий (`user_consent`: `personal_data_processing`, `terms_of_service`), `dataRegion` org (`global` \| `ru`).

### 0.3. Локализация БД (ч. 5 ст. 18 152-ФЗ) — **BLOCKER**

| Факт | Источник |
|------|----------|
| Primary DB = Neon PostgreSQL | архитектура / installation |
| Публично: *«Не заявляем residency в РФ, пока не подключён отдельный RU DB contour»* | `/docs/data-storage` |
| Org default `dataRegion = global` | schema |
| При `dataRegion=ru` блокируются openai / stream / anam / elevenlabs | `assert-foreign-data-processing.ts` |
| Гайд RF-контура есть, но не заявлен как факт продакшена | `docs/DEPLOYMENT_RF.md` |

**Правило плана:** пока primary не в РФ (или нет схемы «первичка РФ → выгрузка»), Политика **не пишет** «хранение на территории РФ» как общее утверждение. Допустимы: (а) честное «Global + опция Enterprise RU», (б) отложенная публикация полного пакета до RU-контура.

→ В финальной версии: выбранный контур на дату публикации + что писать в Политике/DPA § локализация.

### 0.4. Трансграничная передача — **BLOCKER (связано с 0.3)**

Фактические зарубежные процессоры в продукте: LLM (OpenAI/xAI и др.), Anam (avatar/STT/TTS), ElevenLabs (TTS), xAI Voice, платёжные/hosting провайдеры по конфигурации.

Политика § трансграничная передача и DPA § субобработчики пишутся **только после** списка провайдеров + уведомления РКН (или явного «не передаём / только РФ-модели»).

→ В финальной версии: реестр субпроцессоров v1 + статус уведомления РКН.

### 0.5. Биометрия / образ / голос — **OPEN → юрист**

| Пайплайн | Назначение в продукте | Черновая квалификация для плана |
|----------|----------------------|----------------------------------|
| Anam face video | синтез внешности цифрового сотрудника, не face-ID | не биометрия для идентификации пользователя |
| User mic → STT | распознавание речи для диалога | не speaker verification |
| ElevenLabs / xAI TTS | синтез голоса employee | образ/голос **персонажа**, не идентификация клиента |
| Загрузка фото аватара | контент org для employee | ст. 152.1 ГК РФ — согласие носителя образа |
| Landing demo Adeline | публичный показ готовой персоны | отдельный дисклеймер + политика визита |

**В документах AUP/Политики зафиксировать прямо:** платформа **не** использует голосовую/лицевую биометрию для установления личности пользователя; идентификация — по учётной записи.  
**Не решать самостоятельно:** клонирование голоса/лица реального человека клиентом; 572-ФЗ/ЕБС — только если появится идентификация.

### 0.6. НДС и тарифы — **OPEN**

| Факт | Источник |
|------|----------|
| Цены в UI: ₽ строки (`599 ₽ / mo` …) без «вкл. НДС» | `plans.ts`, `/docs/plans` |
| T-Bank receipt VAT default | `TBANK_VAT=none`, `TBANK_TAXATION=usn_income` |
| Реестр российского ПО / аккредитация Минцифры | **не найдены в репо** |
| RU acquiring | в scope как ⬜ |

→ В финальной версии обязательно:  
`VAT_MODE = none | vat20 | exempt_149` + формулировка на витрине тарифов + соответствие чекам Т-Банка.

## 8. Карта «структура → продукт»

### Док. 1 Оферта

Опирается на: SaaS доступ, лицензия неисключительная, акт/УПД (ЭДО TBD), T-Bank / Polar (что реально включено на дату), возвраты, API keys, AI disclaimer, кап ответственности (только при B2B_ONLY).

Жёсткий блок: выдача ИИ за человека, дипфейки, голос/образ, запрет обучения чужих моделей на выводе, КИИ, спецкатегории ПДн. Согласовать с product claim «раскрытие ИИ-природы» (Adeline/Yuki copy).

Только **операторский** контур. Честно: Neon, регион, ИИ-провайдеры, эмбеддинги, STT/TTS, отсутствие face-ID. Таблица сроков. Инциденты 24h/72h. Ответственный — из финальной версии.

### Док. 4 Согласия

Минимум к релизу лендинга+регистрации:

- 4.1 визит / лид (если формы)  
- 4.2 регистрация (уже есть галочка + `recordUserConsents`)  
- маркетинг — отдельная галочка, если появится рассылка  

4.3–4.7 — только если включаем соответствующие функции (распространение ПДн, биометрия, авто-решения, HR, клон образа).

### Док. 5 Cookie

Сверить с фактом: Better Auth session + UI prefs; «нет рекламной аналитики» — только если правда. Баннер — после решения по аналитике (Яндекс vs GA).

### Док. 6–7 Реквизиты / Контакты

ЕГРЮЛ-секция на company; контакты без внешней «ЕГРЮЛ» в списке sales.

### Док. 8 Тарифы

Приложение; цифры только здесь и в Billing UI; оферта — отсылка.

### Док. 9 DPA

Обязателен до Enterprise/банк. Реестр субобработчиков. Локализация/трансграничка — после 0.3–0.4.

### Док. 10 SLA

Сначала измерить uptime; кредиты как единственное средство; исключения: клиент, форс-мажор, сбои model providers, регулятор.
